DOCUMENTACAO TECNICA - AUTOMACAO MONGODB
=======================================

1. OBJETIVO
-----------

Este projeto automatiza a leitura e a execucao de lotes MongoDB para a
operacao de sustentacao B2C.

Existem duas abordagens:

- V1: usa mongosh
- V2: usa pymongo

A V1 foi preservada.
A V2 foi ajustada para manter o fluxo simples da V1, mas executando pelo
driver Python.


2. DIFERENCA ENTRE V1 E V2
--------------------------

V1
- arquivo: scripts/automacao_mongodb_v1.py
- leitura de .txt e .zip
- simulacao do lote
- execucao via mongosh
- o Python nao interpreta a query

V2
- arquivo: scripts/automacao_mongodb_v2_pymongo.py
- leitura de .txt e .zip
- simulacao do lote
- execucao via pymongo
- o Python extrai apenas o minimo necessario da linha para chamar
  update_one ou update_many

Analogia com Java:
- V1 se parece com gerar um script e mandar um cliente externo executar
- V2 se parece com abrir uma conexao JDBC e chamar a operacao direto


3. FORMATO DO LOTE
------------------

Formato padrao suportado atualmente:

db.document.updateMany({"customer.document": "11111111110001", "document.barCode": "000000000000000000000000000000000000000000000001", "document.flProForma": false}, {$set: {"customer.accountNumber": "99999900001"}});

Observacoes:
- um comando por linha
- a linha termina com ;
- o filtro usa:
  - customer.document
  - document.barCode
  - document.flProForma
- o update usa:
  - $set
  - customer.accountNumber


4. COMO A V2 FUNCIONA
---------------------

Passo 1
- o usuario informa um .txt ou um .zip

Passo 2
- o script le o conteudo

Passo 3
- o script mostra simulacao ou segue para execucao

Passo 4
- na execucao, cada linha do arquivo e convertida em:
  - collection
  - metodo
  - filtro
  - update


5. FUNCOES DO PROJETO
---------------------

Esta secao foi organizada para evitar repeticao.

Primeiro aparecem as funcoes comuns entre V1 e V2.
Depois aparecem apenas as funcoes exclusivas de cada versao.


5.1 FUNCOES COMUNS ENTRE V1 E V2
--------------------------------

ler_arquivo(caminho)
- existe nas duas versoes
- recebe um caminho `.txt` ou `.zip`
- se for `.zip`, abre o pacote e junta os `.txt`
- se for `.txt`, le o arquivo direto
- devolve:
  - se deu certo
  - o conteudo do lote
  - os arquivos lidos

simular(conteudo, arquivos)
- existe nas duas versoes
- mostra na tela o lote que seria executado
- nao altera nada no banco
- serve como etapa de revisao antes da confirmacao

menu()
- existe nas duas versoes
- controla o fluxo interativo
- pede o caminho do arquivo
- pergunta se o usuario quer simular ou executar
- pede confirmacao antes da execucao real


5.2 FUNCOES EXCLUSIVAS DA V1
----------------------------

Arquivo:
- scripts/automacao_mongodb_v1.py

executar(conteudo)
- existe apenas na V1
- recebe o lote ja pronto em texto
- adiciona `use smartbill;` no topo do script
- grava o lote em um arquivo temporario `.js`
- chama o `mongosh` via `subprocess`
- mostra a saida do shell
- retorna sucesso ou erro

O ponto principal da V1 e:
- o Python nao interpreta o conteudo do update
- quem entende os comandos e o `mongosh`


5.3 FUNCOES EXCLUSIVAS DA V2
----------------------------

Arquivo:
- scripts/automacao_mongodb_v2_pymongo.py

obter_linhas_comando(conteudo)
- filtra o conteudo lido do lote
- remove linhas vazias
- remove comentarios simples iniciados por `//`
- devolve apenas as linhas que realmente parecem comandos

dividir_argumentos_principais(argumentos)
- separa os argumentos principais do comando
- no caso do update, separa:
  - filtro
  - update
- faz isso sem quebrar nas virgulas internas dos objetos JSON

normalizar_update(update_texto)
- adapta o trecho do update do estilo do shell MongoDB para algo que o `json.loads` aceite
- exemplo:
  - shell: `{$set: {"campo": "valor"}}`
  - JSON lido pelo Python: `{"$set": {"campo": "valor"}}`

validar_estrutura(filtro, atualizacao, numero_linha, arquivo)
- garante que a linha segue o contrato esperado da V2
- verifica se os campos obrigatorios do filtro existem
- verifica se o update usa apenas `$set`
- se a estrutura vier fora do combinado, para a execucao e informa erro claro

parsear_linha(numero_linha, linha, arquivo)
- transforma uma linha do `.txt` em uma estrutura Python pronta para execucao
- identifica:
  - collection
  - metodo (`updateOne` ou `updateMany`)
  - filtro
  - update
- esta e a funcao mais importante da V2 porque faz a ponte entre:
  - texto do shell MongoDB
  - chamada Python do `pymongo`

preparar_comandos(conteudo, arquivos)
- percorre todas as linhas do lote
- chama `parsear_linha` para cada uma
- monta a lista final de comandos que sera executada
- falha cedo se alguma linha estiver fora do formato suportado

executar(conteudo, arquivos)
- existe apenas na V2 com essa responsabilidade
- abre a conexao com `MongoClient`
- seleciona o banco `smartbill`
- para cada linha:
  - escolhe a collection correta
  - executa `update_many` ou `update_one`
- imprime `matched` e `modified`
- retorna sucesso ou erro

O ponto principal da V2 e:
- o Python precisa entender minimamente a linha
- isso e necessario porque o `pymongo` nao executa texto cru do shell
- ele executa chamadas Python com filtro e update separados

Passo 5
- o script abre:
  client = MongoClient(...)

Passo 6
- o equivalente ao "use smartbill" e:
  db = client["smartbill"]

Passo 7
- o script chama:
  - update_many(...)
  - ou update_one(...)


6. DESCOBERTA IMPORTANTE DOS TESTES
-----------------------------------

Durante os testes, os updates pareciam falhar.

Resultado observado no mongosh:

{
  acknowledged: true,
  matchedCount: 0,
  modifiedCount: 0
}

Isso nao significava erro de sintaxe.
Significava que o filtro nao encontrava documentos.

Causa real encontrada:
- o arquivo queries_exemplo.txt usava db.document
- mas a massa inicial do banco tinha apenas:
  - clientes
  - contratos
  - faturas
  - pagamentos
- a collection document estava vazia

Conclusao:
- a automacao estava correta
- a massa nao era compativel com o lote


7. VALIDACAO REALIZADA
----------------------

Ambiente validado:
- container Docker: mongo-teste
- porta: 27017
- banco: smartbill

Foi inserida massa de teste compativel na collection document.

Depois disso, a V2 foi executada contra queries_exemplo.txt com resultado:
- 60 comandos lidos
- 60 comandos encontrados
- matched = 60
- modified = 59

Observacao:
- o primeiro documento ja tinha sido alterado num teste isolado anterior
- por isso ele foi encontrado, mas nao precisou ser modificado novamente


8. COMO DIAGNOSTICAR NO MONGOSH
-------------------------------

Entrar no banco:

use smartbill

Ver collections:

show collections

Ver se a collection existe:

db.document.countDocuments({})

Testar o filtro antes do update:

db.document.countDocuments({
  "customer.document": "11111111110001",
  "document.barCode": "000000000000000000000000000000000000000000000001",
  "document.flProForma": false
})

Interpretacao:
- 0 = nao ha documento para alterar
- maior que 0 = o filtro encontra registro
